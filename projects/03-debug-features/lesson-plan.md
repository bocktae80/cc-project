# 튜터 지시서: 디버그 기능

> 이 파일은 `/learn 03d` 실행 시 튜터(클로드)가 읽는 지시서입니다. 사용자에게 직접 보여주지 마세요.

## 학습 목표

`--debug` 플래그와 `/debug` 커맨드로 클로드 코드의 내부 동작을 들여다보는 방법을 배운다.

## 핵심 비유

"--debug는 자동차의 블랙박스예요 — 모든 걸 기록하죠. /debug는 계기판이에요 — 지금 상태를 한눈에 보여줘요. 문제가 생겼을 때 계기판(/debug)을 먼저 보고, 더 자세히 알고 싶으면 블랙박스(--debug)를 꺼내는 거예요."

## 연습 파일

- 프로젝트 README: `projects/03-debug-features/README.md`

## 교육 흐름

### Step 1: /debug — 계기판 보기

1. "클로드가 지금 어떤 상태인지 궁금할 때가 있죠? /debug를 써보세요!"
2. `/debug` 커맨드가 보여주는 정보 설명:
   - 세션 ID와 대화 턴 수
   - 사용한 토큰 수 (입력/출력)
   - 컨텍스트 윈도우 사용량
   - 로드된 시스템 프롬프트 크기
   - 활성화된 도구 목록
3. "계기판처럼 한눈에 볼 수 있어서 빠르게 진단할 수 있어요"
4. 사용자에게 제안: "지금 바로 /debug를 입력해볼까요? (이 튜터 세션의 상태를 볼 수 있어요!)"

### Step 2: --debug — 블랙박스 열기

1. "더 자세한 기록이 필요하면 --debug 플래그를 써요"
2. 사용 방법 설명:
   - 시작할 때: `claude --debug` (모든 내부 로그를 기록)
   - 로그 위치: `~/.claude/debug/latest`
3. 로그 구조 설명:
   - 타임스탬프: 언제 일어났는지
   - 카테고리: `[init]`, `[API:request]`, `[tool:use]` 등
   - 로그 레벨: `DEBUG` (상세) / `WARN` (경고) / `ERROR` (오류)
4. "블랙박스는 매번 켤 필요 없어요. 문제가 생겼을 때만 켜면 돼요!"

### Step 3: 실전 시나리오

1. 시나리오 제시: "클로드가 이상하게 느려졌을 때"
   - `/debug` 먼저 확인 → 토큰 사용량 체크
   - 컨텍스트 윈도우가 가득 찼다면? → `/compact`로 압축
   - 그래도 느리면? → `--debug`로 시작해서 API 요청 시간 확인
2. 시나리오 2: "도구가 작동하지 않을 때"
   - `/debug`로 활성 도구 목록 확인
   - `--debug` 로그에서 `[tool:use]` 카테고리로 에러 추적
3. 프로젝트 README를 Read로 읽어서 추가 설명 보여주기

### Step 4: 마무리 퀴즈

1. "--debug와 /debug 중 '실시간 상태 확인'에 쓰는 건?"
   → 정답: /debug (계기판 역할)
2. "debug 로그는 어디에 저장되나요?"
   → 정답: ~/.claude/debug/latest
3. "클로드가 느릴 때 제일 먼저 할 일은?"
   → 정답: /debug로 토큰 사용량과 컨텍스트 윈도우 상태 확인
